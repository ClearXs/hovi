# 投标文件生成器

> ⚠️ **【执行说明】只需要用户确认一次！**
>
> - 步骤2.5：用户确认解析结果
> - 步骤4.5：**用户确认"开始生成"后**，自动完成所有后续步骤
> - 步骤6.5：自动执行，不需要用户确认

## 预设配置

当用户未明确指定时，使用以下默认值：

- **目标页数**：150页
- **代码原型**：是（生成HTML原型）
- **图表**：系统架构图、网络拓扑图、功能模块图、甘特图
- **自动扩充**：是（字数不足自动添加内容）
- **多Agent并行**：是（自动使用多Agent加速生成）

---

智能分析招标文件，从知识库获取历史投标参考，自动生成结构化的投标文件（技术标+商务标）和可运行的代码原型，并输出Word文档格式。

## 何时使用

当用户需要制作投标文件时使用此技能：

- 根据招标文件生成完整投标文件
- 解析PDF/URL格式的招标文件
- 从 `{workspace}/招标书/` 目录获取招标文件
- 搜索历史投标文档作为参考
- 生成评分标准应对策略
- 生成技术方案代码原型（Next.js + React + shadcn/ui）

**提示**：系统有定时任务在 `{workspace}/招标书/` 目录下抓取招标书，可以直接使用该目录。

## 工作流程

> **【执行流程概览】**
>
> 1. 步骤1 → 接收招标文件
> 2. 步骤2 → 解析招标文件
> 3. 步骤2.5 → 确认解析结果
> 4. 步骤3 → 搜索知识库和招标书目录
> 5. **【步骤4.5 → 确认投标大纲】← 必须等待用户确认"开始生成"后才能继续！**
> 6. 步骤5 → 生成技术内容（一次性完整生成）
> 7. 步骤6 → 生成代码原型（自动完成，图表用占位符）
> 8. 步骤7 → 生成Word文档（自动完成）
> 9. 步骤8 → 询问用户是否打开文件 + 输出文件清单
>
> **【关键】只有步骤4.5需要用户确认（说"确认，开始生成"），其他步骤自动完成，不需要用户干预！步骤8需要用户确认是否打开文件。**

### 步骤1：接收招标文件

支持以下输入方式：

- **PDF文件**：用户上传招标文件PDF
- **URL链接**：招标网站链接
- **文本内容**：复制粘贴的招标内容
- **{workspace}/招标书/目录**：系统定时抓取的招标文件（优先使用最新或指定的招标文件）

**注意**：系统有定时任务在 `{workspace}/招标书/` 目录下抓取招标书，可以直接使用该目录下的招标文件。

### 步骤2：解析招标文件

使用 LLM 解析招标文件，提取关键信息。

**解析方式**：

1. 如果是PDF文件，先用 `scripts/parse_tender.py` 提取文本内容
2. 如果是URL，用适当工具获取网页内容
3. 将内容发送给 LLM 进行语义解析

**解析Prompt**：

```
请分析以下招标文件内容，提取以下关键信息并以Markdown格式输出：

## 一、项目基本信息
- 项目名称：
- 项目编号：
- 招标单位：
- 预算金额：
- 投标截止时间：
- 开标时间：
- 服务期限：

## 二、资质要求
- 投标人资格条件：
- 资质证书要求：
- 业绩要求：

## 三、评分标准
- 评分办法（综合评分法/最低价法等）：
- 技术评分权重：
- 商务评分权重：
- 各评分因素及得分点：

## 四、技术要求
- 采购需求/服务范围：
- 技术规格/参数：
- 功能要求：
- 交付要求：

## 五、商务要求
- 投标保证金：
- 付款方式：
- 合同期限：
- 其他商务条款：

## 六、招标文件的特殊要求
- 格式要求：
- 份数要求：
- 其他特殊说明：

请尽可能详细地提取以上信息，如果有不确定的内容请标注"待确认"。
```

**输出**：生成 `招标文件解析.md`，包含结构化的招标信息。

**用户确认**：解析完成后，必须展示给用户确认关键信息是否准确。

### 【必须确认】步骤2.5：确认解析结果

**必须与用户确认以下内容**：

1. **项目基本信息**：项目名称、编号、预算等是否正确
2. **资质要求**：是否满足投标资格
3. **评分标准**：评分权重和得分点是否清晰
4. **技术要求**：功能需求是否完整理解
5. **商务要求**：报价和付款方式是否明确

**用户确认后**，解析结果将作为生成投标文档的重要来源。

### 步骤3：搜索知识库 + 网上搜索（必须执行，资料越多越好）

> ⚠️ **【必须执行】此步骤不能跳过！必须搜索尽可能多的参考资料**
>
> - **资料越多，生成的内容越专业**
> - **每个章节至少需要2-3个参考来源**

使用 moltbot 知识库工具搜索历史投标文档，以及从 `{workspace}/招标书/` 目录获取参考文档。

**关键**：必须搜索并参考**同类型、同行业**的真实投标文件案例，让AI学习专业投标文档的风格和结构。

**【第一步】搜索知识库（至少搜索10个相关文档）**：

1. **先从招标书目录获取**：检查 `{workspace}/招标书/` 目录下的招标文件
   - 查看目录下的PDF文件
   - 选择与当前项目相关的招标文件

2. **列出知识库文档**：

```
使用 tool: knowledge_list
参数: { "limit": 50 }
```

3. **搜索相似项目**（必须执行，搜索多个关键词）：

```
使用 tool: knowledge_search
参数:
{
  "query": "投标文件 技术方案 项目名称/行业/系统类型",
  "limit": 10
}
```

**搜索示例**（每个关键词都要搜索）：

- "智慧校园 投标文件 技术方案"
- "智慧校园 信息化 建设方案"
- "智慧校园 系统架构 设计"
- "智慧校园 设备清单 参数"
- "教育信息化 投标文件 案例"

4. **获取文档详情**：找到相关文档后，获取详细内容作为参考

```
使用 tool: knowledge_get
参数:
{
  "documentId": "文档ID"
}
```

**【第二步】网上搜索补充（每个章节至少2-3个参考来源）**：

必须网上搜索更多资料：

1. **每个章节都要搜索相关资料**：
   - 项目概述：搜索"智慧校园 项目背景 建设目标"
   - 需求分析：搜索"智慧校园 功能需求 业务分析"
   - 系统设计：搜索"智慧校园 系统架构 设计方案"
   - 技术方案：搜索"智慧校园 技术选型 实施计划"
   - 实施/质量/售后：搜索对应章节的参考内容

2. **搜索内容要求**：
   - 设备参数表（品牌、型号、规格）
   - 技术方案详细描述
   - 实施计划模板
   - 行业标准规范

3. **整理参考资料**：
   - 每个章节列出参考来源
   - 标记哪些内容来自知识库
   - 标记哪些内容来自网上搜索

**【第三步】整理参考材料**：

- 将找到的知识库内容和网上内容整理成参考资料
- 标记哪些内容来自知识库，哪些来自网上搜索
- 这些参考资料将在生成内容时使用

```
使用 tool: knowledge_get
参数:
{
  "documentId": "文档ID"
}
```

**搜索技巧**：

- 使用项目相关的关键词（行业、系统类型、技术栈）
- 搜索"投标文件"、"技术方案"、"实施方案"等关键词
- 优先选择同行业、类似项目规模的文档

**历史文档用途**：

- 参考投标文件的结构和章节安排
- 借鉴优秀的技术方案描述
- 复用可行的实施计划
- 了解类似项目的评分要点

**注意**：搜索结果仅作为参考，不要直接复制内容，需要根据本次招标要求进行修改。

### 步骤4：生成投标大纲（初稿）

**技术标结构**：

1. 投标函
2. 法定代表人身份证明
3. 授权委托书
4. 投标保证金
5. 资格证明文件
6. **技术/商务偏离表**（正偏离说明 - 优于招标要求的技术参数和服务方案）
7. **项目概述**（项目背景、建设目标、系统定位）
8. **详细需求分析**（需求理解、业务分析、功能需求、非功能需求）
9. **系统总体设计**（技术架构设计、网络架构设计、安全架构设计、数据架构设计）
10. **功能模块详细设计**（每个模块的详细设计、流程图、接口设计）
11. **技术方案**（技术选型、关键技术方案、实施方案）
12. **项目实施方案**（实施计划、进度安排、里程碑、风险控制）
13. **质量保证措施**（质量标准、质量管理、测试方案、缺陷管理）
14. **售后服务方案**（服务内容、响应时间、培训计划、维护方案）
15. **项目团队介绍**（组织架构、人员资质、类似项目经验）
16. **代码原型**（可选）

**商务标结构**：

1. 报价函
2. 报价清单
3. 价格说明
4. 商务响应偏离表

详见 [references/tender-template.md](references/tender-template.md)

### 步骤5：生成技术内容（核心步骤，**一次性生成完整**）

**这是最关键的步骤**，必须**一次性生成所有章节的完整内容**，不能分批生成，不能生成"补充内容"。

> **【关键】禁止分批生成和补充生成**：
> - ❌ 错误的："先生成一部分，后面再补充"
> - ❌ 错误的："这是补充内容，我们继续"
> - ❌ 错误的：生成完后又生成新的内容作为"补充"
> - ✅ 正确的：**一次性生成所有章节的完整内容**
> - ✅ 正确的：生成过程中如果发现内容不足，直接在当前章节扩充

> **【必须使用参考资料】**：
> - 使用步骤3收集的知识库内容作为参考
> - 使用步骤3收集的网上搜索内容作为参考

   - 增加更多功能模块的详细设计
   - 添加更多接口设计文档
   - 增加更多测试用例
   - 扩充设备参数表
   - 增加更多业务流程说明
5. **重复验证，直到达到目标页数**
6. **最终输出实际页数**：实际页数：XXX页（已达标/未达标）

生成Word文档步骤：

1. **使用 md2docx.py 直接从Markdown生成**（不要使用模板！）
   - 脚本会自动从Markdown文件生成Word文档
   - 会自动生成目录
   - 会保留所有格式

2. **内容必须达到用户确认的总页数**（100-400页）
   - **生成后必须验证实际页数**
   - 不足则继续扩充

3. **【必须】添加目录（Outline）**：
   - Word文档必须包含目录
   - 目录在封面之后，正文之前
   - 包含所有章节和子章节标题
   - 包含页码

   ```python
   from docx import Document
   from docx.enum.text import WD_ALIGN_PARAGRAPH

   doc = Document('投标文件.docx')

   # 添加目录标题
   heading = doc.add_heading('目 录', level=1)
   heading.alignment = WD_ALIGN_PARAGRAPH.CENTER

   # 添加目录内容（手动添加各级标题和页码）
   # 格式：
   # 第一章 投标函..........................1
   #   1.1 投标函........................1
   # 第二章 项目概述.......................3
   #   2.1 项目背景......................3
   ```

4. **每个章节必须有内容**：不能跳过任何章节
   - 章节1-5：投标函、法定代表人、授权委托书、保证金、资格证明
   - 章节6：项目概述
   - 章节7：详细需求分析
   - 章节8：系统总体设计
   - 章节9：功能模块详细设计
   - 章节10：技术方案
   - 章节11：项目实施方案
   - 章节12：质量保证措施
   - 章节13：售后服务方案
   - 章节14：项目团队介绍

5. 嵌入图表占位符

6. 根据用户确认的格式填充技术标和商务标内容

7. 添加评分响应表

8. 设置页眉页脚

9. 输出到 `{workspace}/bid-outputs/{项目名称}/投标文件.docx`

**Word文档生成要点**：

- **每个章节**都必须有详细的内容，页数必须达到用户确认的总页数
- **每个子章节**都要有详细论述
- **图表必须嵌入**到对应章节中
- 表格需要格式规范
- 段落需要有缩进和对齐
- 页码需要连续
- **禁止包含**：评分提示、内部说明、非正式表述等无关内容

**嵌入图片到Word**：

```python
from docx import Document
from docx.shared import Inches, Pt
from docx.enum.text import WD_ALIGN_PARAGRAPH

doc = Document('模板.docx')

# 添加图片
doc.add_picture('diagrams/系统架构图.png', width=Inches(6))

# 设置图片居中
last_paragraph = doc.paragraphs[-1]
last_paragraph.alignment = WD_ALIGN_PARAGRAPH.CENTER
```
